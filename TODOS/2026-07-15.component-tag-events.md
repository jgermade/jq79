# `@event` on component tags — wire them to the child's `$emit`

**Status:** decided (sibling of `2026-07-15.model-directive.md`; the fork was
the user's call). Not yet implemented.

## The problem

`@event` attributes on a component tag are silently dropped: the tag renders
as two comment anchors, `renderNestedComponent` skips `@` attrs when
collecting props, and nothing ever hears them. The working pattern is a
listener on a wrapping element (`<div @changed=…><Stepper/></div>`), which
catches both the child's `$emit`s (bubbling CustomEvents) and native DOM
events bubbling out of the child's DOM — but the tag itself, the obvious
place to write the handler, does nothing and says nothing.

## The proposal

`@event` on a component tag subscribes to that instance's `$emit` channel:

```html
<Stepper @changed="last = $event.detail" />
<LoginForm @submit.prevent="handleSubmit" />
```

Wired via `instance.on(name, listener)` in `renderNestedComponent`, per
instance, inside the creation effect — the channel that survives
hot-replace (markers are per render generation) and works detached.

**Scope is the `$emit` channel only.** Native DOM events from the child's
inner DOM (`submit` from its `<form>`) bubble through the child's top-level
nodes to shared ancestors — the tag's anchors are siblings of that path, not
on it. A child that wants its native event heard on its tag re-emits it
(`<form @submit.prevent="$emit('submit')">`); the container pattern remains
the way to hear native events without the child's cooperation. Documented
loudly, because it *will* surprise.

### `$emit` grows teeth for it

For `.prevent`/`.stop` to mean anything, `$emit` is restructured:

```
1. the CustomEvent is created cancelable
2. this instance's on() listeners run first (the tag/API channel)
3. unless one called stopPropagation() (readable as event.cancelBubble),
   the event is dispatched from the start marker as before (the DOM channel)
4. $emit returns !event.defaultPrevented
```

So on a component tag: `.prevent` makes the child's `$emit(...)` return
`false` — a "the parent vetoed" contract that didn't exist ($emit previously
always returned true, the dispatch being non-cancelable); `.stop` keeps the
emit off the DOM entirely (containers above never hear it); `.once`
unsubscribes after the first call. `.self` and `.capture` have no meaning on
this channel and are ignored.

**Behavior change, on purpose:** `on()` listeners used to run *after* the
DOM dispatch; now they run before (they must, to be able to stop it), and
every `$emit` event is cancelable. Nothing in the repo reads the old order
or the return value.

## What it costs

- The `$emit` restructure (~5 lines), a `wireTagEvent` sibling of
  `bindEvent` (~15 lines — same expression/modifier handling, `instance.on`
  instead of `addEventListener`), docs, tests.

## Rejected

- **DOM listeners on the instance's start marker** — the marker *is* the
  dispatch target, so target-phase listeners would work, but markers are
  recreated per render generation (hot-replace would orphan the parent's
  template listeners) and the field is private. `emitListeners` is the
  designed channel.
- **Listeners on the child's top-level nodes** (to catch native events on
  the tag) — that set is dynamic (`:if`/`:each` swap them), and chasing it
  buys what the container pattern already does.
- **Warn-only** (keep dropping, add a console.warn) — the warn without the
  wiring documents a hole instead of filling it, and `:model`'s machinery
  puts the wiring within reach.

## Open
