# `:props="obj"` — spreading an object's properties as props

**Status:** proposal.

## The problem

Passing a bag of props to a child component means naming each one, even when the
parent already holds them together in one object:

```html
<SdkInfo :name="sdk.name" :version="sdk.version" :arch="sdk.arch" />
```

The object is right there (`sdk`), but there's no way to say "pass all of it".
Every component framework grows this: React/JSX `{...props}`, Vue `v-bind="obj"`,
Svelte `{...props}`. jq79 has no equivalent.

`:with="obj"` looks like it should do this, and doesn't — it narrows the render
scope (names resolve against `obj` first, then fall back), so `<Child :with="obj" />`
passes *nothing*: prop collection skips `:with` (it's a `CONTROL_ATTRS` member),
and there are no other prop attributes on the tag. Overloading `:with` to also
inject props would fork one attribute's meaning by tag kind (scope-narrowing on a
plain element, prop-spread on a component), which is why this is a new directive
instead.

## The proposal

`:props="expr"` on a component tag evaluates `expr` and spreads its own
enumerable properties as props to the child, each kept in sync reactively:

```html
<SdkInfo :props="sdk" />                    <!-- name, version, arch, … -->
<SdkInfo :props="sdk" :arch="'arm64'" />    <!-- spread, with one override -->
```

Semantics:

- **Object only.** `expr` must evaluate to a non-null object; anything else
  (undefined mid-`await`, a primitive) spreads nothing — fail closed, matching
  `createWithScope`'s "not an object → no narrowing" rule.
- **Reactive, key set included.** Props today are a fixed set of static
  attributes; a spread's key set is dynamic. One effect reads the object, writes
  each own key onto the child store, and on re-run clears (`= undefined`) keys
  that have disappeared since the last run — so removing a property from the
  object un-sets the prop, and adding one sets it.
- **Precedence is source order**, mirroring JS object spread. Attribute order
  survives to the AST (`Array.from(el.attributes)` → `Object.fromEntries`, both
  insertion-ordered), so `:props="obj" :name="x"` means `{ ...obj, name: x }`
  (explicit wins) and `:name="x" :props="obj"` means `{ name: x, ...obj }` (obj
  wins). See Open — this is the one reversible call worth a second look.
- **Multiple spreads** compose: `:props.0="a" :props.1="b"` (the numeric suffix
  only makes the attribute names distinct so `Object.fromEntries` doesn't dedupe
  them). Later wins, again by source order.

### Sugar: `...expr`

`...expr` as an attribute is nicer to read and is *pure sugar* — a pre-DOM string
rewrite into `:props.<n>="expr"`, so the runtime only ever knows `:props`:

```html
<SdkInfo ...sdk />                 <!-- → <SdkInfo :props.0="sdk" /> -->
<SdkInfo :arch="'arm64'" ...sdk /> <!-- → <SdkInfo :arch="'arm64'" :props.0="sdk" /> -->
```

The rewrite has to run *before* DOMParser and must capture the expression from
the **raw source**, because the HTML parser lowercases attribute *names*: with
the expression living in the name (`...userData`), `userData` → `userdata` and
the prop silently resolves to nothing. Moving it into a *value* (which the parser
leaves untouched) is the whole point — `...userData` → `:props.0="userData"`,
case intact. This is the same "transform the string before parsing" move that
`expandSelfClosingTags` already makes, and it reuses that pass's two defenses
plus one more:

1. split on `RAW_BLOCK_RE` so `<script>`/`<style>` bodies (`const [x, ...rest] =`)
   are never touched;
2. an open-tag regex so only *inside* a start tag is rewritten — text between
   tags (`carga...espera`) is safe;
3. within the tag, an alternation that consumes quoted values as whole units, so
   a genuine JS spread in a *value* (`@click="f(...args)"`, `:x="{ ...a }"`) is
   skipped and only a spread in *name position* rewrites.

Verified against all four on 2026-07-23 (scratch script): `...data ...userData`
→ `:props.0="data" :props.1="userData"`, and each of the text / handler-arg /
object-in-value / script cases left untouched.

`...expr` accepts an **identifier or member path** (`...sdk`, `...props.user`) —
the char class stops at a paren, so `...getProps()` won't rewrite; call
expressions use the value form `:props="getProps()"`. Documented, and cheap to
warn on if a `...` is followed by something the pattern won't take.

## What it costs

- **`isControlAttr` learns `:props`.** Same move `:class.` already made — the
  prefix can't live in the exact-match `CONTROL_ATTRS` set. Otherwise `:props`
  falls through to a literal string attribute / camelCased prop.
- **`renderNestedComponent` gains a spread step.** Today it builds `props`
  (name → expression) from attributes, seeds the child once, then one effect per
  prop keeps it synced ([jq79.ts:355-368](../src/jq79.ts#L355-L368)). Spread adds:
  a helper to evaluate a `:props*` expression to an object, the seed merges spread
  entries in attribute order with explicit props, and one extra effect diffs the
  spread's key set across runs (clearing vanished keys, skipping any key an
  explicit prop owns so precedence holds). ~15 lines.
- **The `...` rewrite** — a new pre-DOM pass (~10 lines) with its own test table
  (the four corruption cases above, multiple spreads, member paths, a call
  falling through to the value form). Optional: `:props` can ship without the
  sugar and gain it later.

## Rejected

- **Overloading `:with` on component tags.** One attribute, two meanings decided
  by tag kind — and it leaves the interaction with the scope it *also* narrows
  undefined. A separate directive keeps `:with` meaning exactly one thing
  everywhere.
- **`...data` parsed directly as an attribute name** (no pre-DOM rewrite). The
  parser lowercases names, so only all-lowercase expressions survive; `...userData`
  breaks in silence. The whole reason the sugar rewrites into a *value* is to
  dodge this.
- **Explicit-always-wins regardless of order** (warn on collision, like `:model`
  vs `:prop`). Simpler to state, but it throws away the source-order information
  the AST already preserves, and it reads wrong next to `...`: someone writing
  `...defaults :name="x"` expects the JS `{ ...defaults, name: x }` override.
  Source order gives both spellings a meaning instead of making one a warning.
- **A spread that also pulls non-own / inherited keys.** Own enumerable only,
  matching object spread and what the store tracks (`isPlainData`).

## Open

- **Precedence direction.** Source order is the proposal and matches JS, but it's
  the one reversible call — if real use shows people expect explicit props to win
  no matter where they sit, that's a rule change, not a rewrite. Worth watching
  before it's load-bearing.
- **`...getProps()`.** Left to the value form for now. If member-with-call paths
  turn out common, the rewrite's expression grammar can grow to balance parens —
  additive, not a break.
- **camelCase in `...expr`** is preserved (it's in the raw source), unlike every
  *dotted* directive (`:class.`, `:model.`, `@event.`) whose name the parser
  lowercases. So `...userData` works where `:class.isActive` doesn't — worth a
  docs note so the asymmetry doesn't read as a bug.

## 2026-07-23 — implemented

Shipped as proposed, source-order precedence included. `isControlAttr` learned
`:props` (exact) and `:props.` (prefix); `renderNestedComponent` collects a
source-ordered `sources` list (named props *and* spreads) alongside the existing
`props` map, and a `resolveProps()` folds it in order — named prop sets one key,
spread `Object.assign`s an object's own enumerable properties, `:model` bindings
applied last so they still win. Seed is `untracked(resolveProps)`.

Sync splits on whether any spread is present: **no spread keeps the old
per-prop effects untouched** (one effect per prop, fine-grained), so nothing
about existing components changed. **With a spread, one merged effect** re-runs
`resolveProps`, writes every resulting key, and clears (`= undefined`) keys a
spread has dropped since the last run — named props are always in the merge so
they're never cleared. The merge-in-one-effect is what makes positional
precedence deterministic: a key set by two sources can't be resolved across
independent effects (whichever re-ran last would win). The cost — a sibling prop
change re-forwards object props to the child — is confined to spread-using tags,
and object re-forwards are the store's intended "deep touch" channel anyway
(reactive.ts set, the same-reference-stays-loud rule).

The `...expr` sugar is `expandPropsSpread`, a pre-DOM pass mirroring
`expandSelfClosingTags`: split on `RAW_BLOCK_RE`, rewrite only inside start tags
(`OPEN_TAG_RE`), skip quoted values (`ATTR_SPREAD_RE`'s string alternates),
emitting `:props.<n>="expr"` with a per-tag counter. Runs *before*
`expandSelfClosingTags` so `<Card ...opts />` desugars then self-closes.

**Discovered constraint (not foreseen in the proposal): a bare `:props="x"` can
appear only once per tag.** Two `:props` attributes are identical *names*, and
the HTML parser keeps only the first — so multi-spread needs distinct names,
which is exactly what the sugar's `:props.<n>` suffix provides. `...a ...b`
works; `:props="a" :props="b"` silently drops the second. This turns the
per-tag `<n>` counter from a nicety into the mechanism that makes multiple
spreads possible at all. The value form stays single-spread; reach for `...` (or
hand-written `:props.0`/`:props.1`) when composing several. Documented under
`:props` in `docs/template-syntax.md`.

Tests (`tests/Component79.test.ts`): spread of own properties, liveness on
mutate, key-appear and key-disappear, explicit-before-spread and
spread-before-explicit precedence, multi-spread via sugar, fail-closed on a
non-object (and spreading once it resolves), `...camelCase` surviving the parser,
and sugar-plus-explicit source order — plus a non-corruption test that a literal
`...` in text, a JS rest in a script, and a `[...items]` spread in a handler
value are all left alone.

Precedence direction stays the open call it was flagged as; nothing here makes
it harder to reverse (it's contained in `resolveProps`'s fold order).
