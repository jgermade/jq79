# `:value` / `:checked` / `:selected` — form-state directives

**Status:** decided (see the decisions appendix in
`2026-07-15.weak-points-audit.md`). Not yet implemented.

## The problem

`:attrs="{ value }"` writes the *attribute*, and a form control's attribute
is only its default: once the user types, the property detaches and later
store writes stop reaching what's on screen (pinned by the audit's tests).
The workaround — `@input` handlers both ways — can't express "the store is
the source of truth for this input".

## The proposal

Three directives that write the DOM *property*, reactively:

```html
<input :value="name" @input="name = $event.target.value">
<input type="checkbox" :checked="agreed" @change="agreed = $event.target.checked">
<option :selected="lang === 'es'">es</option>
```

- `:value` coerces with `String(v ?? "")` and skips the write when the
  property already holds that string — an unguarded write would move the
  caret of the very input the user is typing into on every unrelated
  re-run.
- `:checked` / `:selected` coerce with `!!v` and assign unconditionally
  (boolean property writes don't disturb anything).
- One-way, store → DOM. The way back stays an explicit `@input`/`@change`,
  as in the forms lesson. On `<select>`, `:value` selects the matching
  option, which is how the property already behaves.
- The three join `CONTROL_ATTRS`: like `:class`/`:text`, they are not props
  on a nested-component tag.

## What it costs

- Three entries in `CONTROL_ATTRS` and one small effect block in
  `renderNode`, next to `:class`.

## Rejected

- **Special-casing `value`/`checked`/`selected` inside `:attrs`** — the
  audit's original rejection stands: `:attrs` keeps its one rule ("entries
  become attributes"). Dedicated directives add the property path without
  making `:attrs` ambiguous.
- **`:model` (two-way) in this change** — deferred by the user. If it
  comes, it builds on these: `:model="name"` is `:value` plus the `@input`
  writeback, with its own decisions (checkbox groups, selects, IME).

## Open

- Nothing known.

---

## 2026-07-15 — implemented

As proposed, the same day, with one addition the proposal missed: the
effects register *after* the element's children render, because `:value` on
a `<select>` can only pick an `<option>` that already exists — with the
registration where `:class`'s is (before children), the first run found an
empty select and the initial value silently didn't apply. Tests cover the
typed-into input staying driven (the contrast with `:attrs`), null→`""`,
checkbox re-assertion after a user untick, `<select>` initial + reactive
value, and `:selected` per option. Documented in
`docs/template-syntax.md` next to `:class`.
