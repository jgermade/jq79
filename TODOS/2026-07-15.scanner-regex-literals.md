# The scanners can't see regex literals

**Status:** proposal. The test battery is in (tests/transform.test.ts, four
`it.fails` marking the breakage); the fix is not.

## The problem

Every scanner in `transform.ts` (`transformSetupScript`,
`transformFactoryScript`, `findStatementEnd`, `indexOfTopLevel`,
`splitTopLevel`, `findExportDefault`) skips strings and comments and counts
bracket depth — and has no notion of a regex literal. A `/` is only ever
checked against `//` and `/*`. Two consequences, both empirically pinned in
tests/transform.test.ts ("regex literals: the scanner's blind spot"):

- A literal whose text puts two slashes side by side — `split(/\//)`,
  `/^https:\/\//` — reads as a **line comment**: everything to the end of the
  line is skipped *without being counted*. If the skipped tail is
  bracket-balanced and the statement really did end at that line, nothing
  breaks (the common URL matcher works by luck, and the test says so). If the
  tail hides a closing bracket, the depth sticks high and the damage cascades:
  a `$:` statement swallows the lines after it into its effect; a later `let`
  is no longer "top level" and silently stops being reactive; an
  `export default` is never found and the factory falls back to setup mode.
- Brackets *inside* a regex are counted as code: `split(/[(]/)` inflates the
  depth by one, permanently, with no comment lookalike involved.

The failure that matters is the cascade: once depth is wrong, every
depth-guarded decision after it is wrong too, to the end of the script.

## Can a blank line (or something like it) bound the damage?

Proposed in review discussion, and worth spelling out why not as a hard rule:
**a blank line legitimately lives inside brackets** — a multi-line object
literal with a blank line between groups of keys is ordinary style, and the
battery pins it ("keeps a blank line inside an open bracket from ending the
statement"). Ending statements at blank lines would trade a rare breakage for
a common one.

The same holds for the other candidate signals:

- *A line starting with `let`/`const`/`$:`* — statements do start there, but
  also inside any multi-line callback body (`$: handler = () => {\n  let x
  ...`), which is depth ≥ 1. Legitimate.
- *A `}` that closes more than was opened* — already handled (depth clamps
  at 0 in the main loops).

All of these share one flaw: they only mean "statement boundary" **at depth
0, and depth is exactly the thing the bug corrupts**. A recovery heuristic
can't tell corrupted depth from an honest multi-line literal, so any signal
strong enough to recover from the bug is strong enough to break correct code.
Bounding the damage is the wrong shape; not corrupting the depth is the fix.

## The proposal: skipRegex, gated by the last token

The classic lexer answer, sized to this codebase. A shared `skipRegex(src,
start)` — same family as `skipString` — that consumes a regex literal:
backslash escapes, and `[...]` character classes where an unescaped `/`
doesn't close the literal (the battery pins `/[/]/` working; a naive skip
would break it).

When the scanner meets a `/` that isn't `//` or `/*`, it decides regex vs
division by the **last meaningful token**, the same call every
tokenizer-based highlighter makes:

- After something that *completes* an expression — identifier, number, string
  or template, `)`, `]`, `}`, postfix `++`/`--` — it's **division**: leave it
  as code (today's behavior).
- After anything else — `= ( [ { , ; : ! ? & | ^ ~ + - * / % < >`, a line
  start, or a **reserved word** (`return`, `typeof`, `case`, `in`,
  `instanceof`, `new`, `delete`, `void`, `do`, `else`, `yield`, `await`) —
  division is impossible (it needs a completed expression on its left), so
  it's a **regex**: skip it whole.

### Why this strictly reduces failures

The heuristic can only err by calling a real division a regex — and it can't,
because every "regex position" above is a position where division is a
SyntaxError anyway. The keyword list is reserved words only, deliberately:
`of` is *not* reserved (`const of = 4; of / 2` is legal division), so `of`
stays off the list and `for (x of /re/)` — iterating a regex, which is
nonsense — stays unfixed rather than risking real code.

What remains unfixed, knowingly: a regex directly after `)` or `}` (an
`if (cond) /re/.test(s)` statement header). Indistinguishable from division
without a parser, absurd as a statement, and it fails exactly as it does
today — no regression, just no rescue.

## What it costs

- One `skipRegex` helper (~15 lines) plus a "last token" tracker in each
  scanner loop. The scanners already track `atStatementStart` with the same
  kind of one-character bookkeeping; this generalizes it slightly.
- Six call sites. They share the string/comment skipping by convention rather
  than by abstraction, so this follows the house pattern (duplicated small
  scanners over a shared lexer object).

## Rejected

- **Blank lines (or line-start keywords) as recovery delimiters** — see
  above: every such signal is only valid at depth 0, which is the corrupted
  quantity. Pinned counterexamples in the battery.
- **A real tokenizer/parser** — the project's stated line is "no full JS
  parser, just a scanner", and the runtime ships to the browser; the
  heuristic gets the same answer everywhere it's decidable.
- **Documenting workarounds only** (`new RegExp("...")`, `[/]` instead of
  `\/`) — they work, but two of the four failure modes are *silent* (a
  non-reactive `let`, a swallowed statement), which is the worst kind of
  limitation to leave in a scanner whose whole job is to be forgettable.

## Open

- Whether `findStatementEnd`'s trailing-operator contract (`a +\n  b` ends at
  the newline; chains are written leading-dot) should stay as is. The battery
  pins today's output — including the ugly detail that a comment after the
  dangling operator swallows the injected `});` — but that's a separate
  decision from regex awareness.

## 2026-07-15 — implemented

Shipped as proposed: `skipRegex` (escapes, character classes, flags) plus
`regexAllowed`, the last-token gate, in `transform.ts`. The gate looks
*backward* from the `/` — skipping whitespace and block comments — rather than
tracking tokens continuously: it's only consulted for the rare `/` that isn't
`//` or `/*`, so the scanners pay nothing on the common path and none of their
loops had to restructure.

Wired into seven scan sites: `findStatementEnd`, the `transformSetupScript`
and `transformFactoryScript` main loops, `indexOfTopLevel`, `splitTopLevel`,
`findExportDefault`, and the two bracket-matching loops in `parsePropsPattern`
and `firstParameterSource` (a prop default can be a regex — `{ sep = /[,:]/ }`
used to split at the comma inside the class).

Two decisions the proposal didn't spell out:

- **`skipRegex` stops at a newline.** A regex literal can't contain an
  unescaped one, so hitting it means the classification was wrong or the input
  malformed — stopping there bounds any conceivable misfire to a single line.
  (This is the sound version of the "bound the damage" instinct from the
  review discussion: the bound lives inside the token, where it's a fact of
  the grammar, not at the statement level, where it would be a guess.)
- **The lookback can cross a newline into a line comment's tail** (`a // see
  return` on the line above a line-starting `/`). Getting that right needs
  backward line-comment detection, which needs backward string detection —
  a rabbit hole for a case that requires a comment ending in a reserved word
  directly above a line-starting slash. Left unrescued, like `if (x) /re/`.

The four `it.fails` flipped and were promoted to plain `it`; the seven
boundary pins (division in all forms, `/[/]/`, blank lines in braces, nested
templates, strings, trailing operators) survived untouched, and four new tests
cover the reserved-word gate, flags, division-after-comment and the regex
prop default.
