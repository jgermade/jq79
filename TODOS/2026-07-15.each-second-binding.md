# `:each` second binding: a named index, and object entries

**Status:** proposal. Nothing implemented.

## The problem

`:each` takes exactly one binding — `item in items` — and iterates arrays
only. Two things fall out of that:

- **`$index` doesn't nest.** In `:each` inside `:each`, the inner `$index`
  shadows the outer one, and there is no way to name either. A grid can't
  address its own row number from a cell.
- **Dictionaries can't be iterated.** A store often holds keyed data
  (`{ es: "Hola", en: "Hello" }`); rendering it today means converting to an
  array in the setup script, which detaches the render from the store's own
  add/delete reactivity — the `deleteProperty` trap that just landed
  (`2026-07-14.delete-reactivity.md`) notifies key removal, and `:each`
  can't hear it for want of a way to iterate the object.

## The proposal

A second, optional binding, in the two spellings every template language
converges on (parens optional, Vue-style):

```html
<li :each="item, i in items">{{ i }}: {{ item.name }}</li>
<li :each="(value, key) in props">{{ key }} = {{ value }}</li>
```

- **Arrays**: the second binding is the index — `$index` with a name, so
  nested loops can hold both. `$index` stays (it's in every existing
  template and the docs' own example).
- **Plain objects**: iterate `Object.entries` in insertion order; the second
  binding is the property key, `$index` is the position. Reading the entries
  happens through the store proxy, so each value is tracked per-key: adding a
  key re-renders (the new-key sweep), deleting one now notifies (the new
  trap), and changing a value wakes exactly that entry.
- **Default diff key**: arrays keep position (unchanged); objects use the
  *property key* — the natural stable identity, so deleting one key never
  rebuilds its neighbors. `:key` overrides either, and can reference both
  bindings.
- Anything that is neither an array nor a plain object keeps rendering
  nothing.

## What it costs

- `EACH_PATTERN` grows one optional group and optional parens.
- `renderEach` normalizes both sources to `[at, item]` pairs and defines one
  more scope var. The bucket diff, the ranges and the reuse rules are
  untouched — an object entry is just an entry whose default key is a string.

## Rejected

- **A third binding** (Vue's `(value, key, index)`): `$index` already serves
  the position; a third name can be appended later without breaking the two.
- **Number ranges** (`n in 5`): write `Array.from({ length: 5 })` — one more
  special case in the pattern buys one saved call.
- **Maps/Sets/iterables**: the store doesn't wrap them (`isPlainData`), so
  their contents aren't tracked and an `:each` over them would go silently
  stale — worse than not supporting them. Reconsider if the store ever
  learns collections.

## Open

- Nothing known. `value, key` order (value first) follows Vue; Svelte and
  JS's own `entries()` disagree with each other anyway, so there is no
  neutral choice — picking Vue's matches the audience this syntax borrows
  from.

---

## 2026-07-15 — implemented

As proposed, the same day. `EACH_PATTERN` grew the optional group and
parens; `renderEach` normalizes both sources to `[at, item]` pairs and the
default diff key became `at` — which for arrays is the index it always was,
and for objects is the property key. The bucket diff, ranges and reuse rules
needed no changes; on reuse the named binding is refreshed alongside
`$index`.

Tests cover the named index (fresh after keyed reorders), nested loops
holding both indices, object entries with the parens form, per-key
reactivity including deletes (the `deleteProperty` trap from
`2026-07-14.delete-reactivity.md`, doing exactly what it was built for),
DOM-identity preservation when a sibling key is deleted, and the malformed
`item, in items` still landing on the comment placeholder.
`docs/template-syntax.md` documents both forms.
