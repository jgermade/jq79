# Hardening the effect runner: reentrancy, no-op writes, the $index seam

**Status:** proposal. The breakage is pinned as four `it.fails`
(tests/renderComponent.test.ts ":each flaky seams", tests/reactive.test.ts);
nothing implemented.

## The problems (all confirmed empirically, 2026-07-15 review)

1. **Effects re-enter themselves.** `notify` runs matching effects
   synchronously and unguarded. A store write *during* an effect's own run —
   concretely: an item of a `:each` creating a new store key while it
   renders, which in real code is a child component's setup writing to a
   shared store — triggers the new-key sweep, which re-runs the list effect
   **while it is mid-diff**. The inner run completes and positions its rows;
   the outer run resumes on stale locals and positions its own: rows come out
   duplicated or tripled, and the orphans are zombies with dead effects.
2. **Writes that change nothing still notify.** The set trap doesn't compare
   values, so `x = x` (a normalizing assignment, a prop sync writing the same
   value) notifies — and an effect that writes what it just read wakes itself
   forever. Today that ends in a stack overflow; it also multiplies the
   cascades of problem 1.
3. **The untracked-scope seam.** `$index` (and `:each`'s named binding) are
   plain scope vars — untracked by design. A binding that reads *only* them
   has no deps, so a keyed reorder leaves it stale... until any unrelated
   new-key sweep re-runs everything and "heals" it. Broken-until-something-
   else-happens: the textbook flaky shape.

## The proposal

### 1. A reentrancy guard with a trailing re-run

An effect that receives a notify while running doesn't re-enter: it marks
itself dirty and re-runs *after* the current run finishes, in a loop, until
clean. Synchronous semantics are preserved — no scheduler, no microtask, no
flush timing (the absence of which is a real strength of this design; tests
stay await-free). The trailing run executes against settled state, so the
`:each` diff sees a consistent world and reuses everything it just built.

Runaway protection: a self-perpetuating effect (`x = x + 1` against its own
dep) used to die by stack overflow; with the loop it would freeze instead.
Cap the trailing iterations (100) and `console.error` — the same call Vue
makes ("maximum recursive updates").

### 2. Skip the notify when the value didn't change

`Object.is(previous, next)` in the set trap, checked before assignment —
**except for new keys**, whose whole point is the sweep (`store.x =
undefined` on an absent `x` must still announce the key). `NaN` writes settle
(`Object.is(NaN, NaN)`), `-0`/`+0` still notify, bridges are untouched (the
same nested store re-bridging was already a no-op). This is what every
fine-grained runtime does; it turns problem 2's forever-loop into a settled
no-op and thins the cascades that feed problem 1.

### 3. Refresh a reused entry's effects when its position changed

`EffectScope` gains `refresh()`: re-run every effect registered on that
scope (the runs are already closures at hand; deps are left as they were).
`renderEach` collects reused entries whose `$index` changed and, once
positions are settled, refreshes each — wrapped in `untracked`, so those
reads don't leak into the list effect's own deps. Bindings that track item
properties re-run off the list notification as they always did (the refresh
is a second, idempotent evaluation for them); bindings that track nothing —
the `{{ $index }}`-only case — finally hear about their move.

## What it costs

- Reordering a keyed list evaluates a moved entry's direct bindings twice
  (once from the notification, once from the refresh). Bindings are
  re-runnable by contract; the work is linear in moved entries.
- The no-op-write check is one `Object.is` per set.

## Rejected

- **Batching/microtask scheduling** for problem 1: fixes reentrancy by
  changing the whole programming model (DOM no longer updated when an
  assignment returns), breaks every synchronous test, and imports the flush
  ordering questions jq79 deliberately doesn't have.
- **Backing `$index` with the store** (a hidden bridged key per `:each`):
  leaks into `Object.keys(data)`, `$onAny`, hot-reload snapshots. The seam
  is better paid for at reorder time than carried on the data forever.
- **Dropping (instead of queueing) a notify that lands mid-run**: the write
  is real; ignoring it leaves the effect stale against its own deps — a new
  flavor of problem 3.

## Open / accepted residue

- A dep-less binding *inside a conditional branch* of a moved entry sits on
  the branch's own scope, which `refresh()` doesn't reach (the chooser is
  reached, and re-chooses correctly). Edge-of-edge; revisit if anyone hits it.
- A trailing re-run queued on an effect that is disposed before it drains
  executes once against detached DOM — idempotent and invisible, accepted to
  keep the guard flag-free.

---

## 2026-07-15 — implemented

All three landed; the four `it.fails` flipped to plain `it`. Two things the
proposal didn't know:

- **The no-op-write skip broke the suite as proposed, and the failure taught
  the real contract.** Two tests went red: a nested component stopped seeing
  the parent's deep mutations, and the tutorial's highlight layer froze. Both
  for the same reason: a parent's prop-sync effect forwards `user.name = x`
  to the child by re-assigning the *same* `user` reference into the child's
  store — and the child's listeners live on the child's store, so that
  "write that changes nothing" was the only way the news crossed the store
  boundary. **Same-reference object writes are a load-bearing channel — the
  deep touch — and must stay loud.** The skip now applies to primitives and
  functions only, where same value genuinely means nothing happened. Pinned
  from both sides in tests/reactive.test.ts ("does not notify a primitive
  write that changes nothing" / "keeps a same-reference object write loud").
- **The docs described the old failure mode.** A self-waking effect used to
  recurse until the stack blew; with the guard it would have looped forever
  silently, so the cap (100 rounds + console error) isn't an optimization,
  it's what keeps the failure loud. `setup-scripts.md`, `development.md` and
  `AGENTS.md` now describe the capped behavior.

The refresh path landed as sketched: `EffectScope.refresh()` re-runs the
scope's own effects; `renderEach` refreshes reused entries whose `$index`
changed, after positions settle, under `untracked`.
