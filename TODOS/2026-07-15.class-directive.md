# `:class` — reactive classes on top of the static `class`

**Status:** proposal. Nothing implemented.

## The problem

Toggling a class is the single most common dynamic binding, and today it
spells like this:

```html
<button :attrs="{ class: 'btn' + (active ? ' btn-active' : '') }">go</button>
```

String concatenation for what is conceptually a set of flags — and it
clobbers the static `class` attribute, so the static part has to move into
the expression too. Every template language ends up with a dedicated class
binding for exactly this reason.

## The proposal

A `:class` directive that *adds to* the static `class` attribute instead of
replacing it, accepting the three shapes Vue normalized a decade ago:

```html
<button class="btn" :class="{ 'btn-active': active }">go</button>
<div :class="theme"></div>
<div :class="[theme, { active }]"></div>
```

- **string** — split on whitespace, each token a class.
- **object** — keys whose values are truthy become classes; a key may hold
  several space-separated names.
- **array** — entries normalized recursively (strings and objects mix).
- anything else (`null`, `undefined`, `false`, numbers) contributes nothing,
  so `:class="cond && 'active'"` works.

One effect per binding: it re-evaluates the expression, removes the classes
*it* added last run that are gone now, and adds the new set via `classList`.
Only classes the binding added are ever removed — the static `class` list is
never touched, even when the expression names one of its classes and then
drops it (`class="btn" :class="{ btn: cond }"` keeps `btn` when `cond` goes
false, matching Vue's merge semantics).

Reactivity falls out of the store: the expression's own reads are tracked by
`evalExpr`, and the object form reads each value through the proxy, so
toggling one flag wakes exactly this effect.

## What it costs

- `":class"` joins `CONTROL_ATTRS` (skipped by the static-attr loop and by
  nested-component prop collection).
- A ~6-line `classNames` normalizer and a ~10-line effect in `renderNode`,
  next to the `:attrs` one.

## Rejected

- **Merging into `:attrs`** (special-casing its `class` key): `:attrs` is
  wholesale by design — each run removes what it set and writes the new
  object. A merge rule for one magic key would make its one behavior two.
  Instead: `:class` and `:attrs="{ class }"` on the same element is
  documented as "don't" — `:attrs` re-runs rewrite the attribute and wipe
  what `:class` added until it happens to re-run.
- **`:class` as a prop on nested components** (Vue's fallthrough-to-root):
  jq79 components have no single root to fall through to. On a component tag
  `:class` is ignored, same as `:text`/`:html`; a component that wants a
  class takes it as an ordinary prop under another name.
- **`:style` in the same change**: the same shape (object → inline styles)
  and a natural companion, but it's its own design (units, camelCase vs
  kebab, custom properties) — a separate proposal if wanted.

## Open

- Nothing known.

## 2026-07-15 — implemented

As proposed, the same day. `":class"` joined `CONTROL_ATTRS`; `classNames`
normalizes the three shapes (recursing for arrays, and for object keys, so a
key holding `"a b"` yields two tokens — `classList.add` throws on tokens with
whitespace, so the split is load-bearing, not cosmetic); the effect in
`renderNode` sits next to the `:attrs` one and removes only what it added
last run, skipping names in the static set.

Tests cover the string form (stale classes removed on swap), the object form
merging with a static `class`, the static class surviving being named-then-
dropped by the expression, mixed arrays, the null/number/false no-op,
per-key tracking of a nested store flag, multi-name object keys, and
`:class` on a `:each` item. `docs/template-syntax.md` documents the three
shapes and the two "don'ts" (`:attrs` class key, component tags).

Later the same day, the tutorial picked it up: the attributes lesson
(`01-basics/04`) teaches it and its solution toggles a `busy` class on the
status line, and the lists lesson's solution (`01-basics/03`) dropped its
`:attrs="{ class: todo.done ? 'done' : '' }"` ternary — the exact pattern the
directive exists to replace — for `:class="{ done: todo.done }"`, with the
snippet given in its README since that lesson runs before attributes are
taught. Both behaviours are asserted in `tests/tutorial.test.ts`.
