# Weak-points audit: what the tests didn't pin, and what they now do

**Status:** the tests landed (this same day). The proposals at the bottom are
open — none implemented.

## The problem

Line coverage sat at 98.6% and every suite was green, but coverage measures
what *executes*, not what is *asserted*. An adversarial pass over the four
`src/` modules found a dozen behaviors nobody held with a test: some
security-posture decisions, some silent user traps, some hardening that had
never been exercised. Each was probed empirically first (a scratch test file
against the real runtime) so the tests assert observed behavior, not guesses.

## What the probes found (and the tests now pin)

- **Scanner and destructuring** (`transform.test.ts`, `Component79.test.ts`):
  `let { a } = obj` at a setup script's top level keeps its keyword — the
  binding is local, invisible to the template, not reactive, and *silent*.
  `let a = 1, b = 2` rewrites only `a`; `b` reaches the store by the new-key
  sweep, working by accident. Nested template literals survive `skipString`
  by backtick parity, not by design. All pinned.
- **Effect-runner cutoff** (`reactive.test.ts`): the 100-cycle giving-up
  path had zero coverage. Confirmed the AGENTS.md claim precisely: a
  read-and-write effect settles its *first* run (deps commit after the run)
  and only loops when something external wakes it — then cuts off at 100
  with the console.error, and the triggering assignment still returns.
- **Dot-path collision** (`reactive.test.ts`): a literal key `"a.b"`
  notifies under the same dotKey as nested `a.b`; effects wake spuriously.
  Pinned as a documented limit, not endorsed behavior.
- **`arr.length = 0`** (`reactive.test.ts`): works — index reads track the
  ancestor `"list"`, which overlaps `"list.length"`. (The audit expected
  this to be broken; it isn't. The test keeps it that way.)
- **In-place `sort()` vs sorted-copy reassignment**
  (`renderComponent.test.ts`): in-place sorting swaps through the proxy, and
  mid-swap the array holds one item twice — the duplicate-`:key` degradation
  fires (warn, positional pairing) and rows are rebuilt, losing input state.
  Reassigning `[...users].sort(...)` moves the existing rows. Both pinned;
  the difference belongs in the docs (open, below).
- **`:attrs` falsy values** (`renderComponent.test.ts`): only
  `null`/`undefined`/`false` remove an attribute — `0` and `""` *set* it,
  and a set boolean attribute is ON: `disabled: items.length` disables the
  empty-list button. Pinned as-is.
- **Attribute vs property on inputs** (`renderComponent.test.ts`):
  `:attrs="{ value }"` writes the attribute; once the user types, later
  store writes stop reaching what's on screen. Pinned.
- **`:class` + `:attrs` class key** (`renderComponent.test.ts`): the
  documented "don't" — an `:attrs` re-run rewrites the whole attribute and
  wipes `:class`'s additions until it re-runs. Pinned as degradation.
- **No component recursion by construction** (`renderComponent.test.ts`):
  the audit went looking for a stack overflow and found the opposite — a
  component tag inside its own template renders one level and stops, because
  prop names arrive HTML-lowercased and can never be the PascalCase key a
  component tag needs. `<A>` inside A falls back to a plain `<a>` anchor.
- **`</script>` inside a script string** (`Component79.test.ts`): the block
  truncates at the first `</script>` (as in any browser) and the mount
  throws a synchronous SyntaxError. Loud, at least. Pinned.
- **Sanitizer evasion corpus** (`dom.test.ts`): math/svg mutation vectors
  and the noscript raw-text escape die at the tag allowlist (dropped whole);
  entity-encoded and tab-split `javascript:` hrefs die at the parsed-protocol
  check; uppercase tags/attrs normalize; `data:` in img src rejected; 500
  levels of nesting survive. Protocol-relative `//host` is *allowed* —
  pinned as a decision (it resolves under the page's scheme).
- **`:html` + `<style scoped>`** (`Component79.test.ts`): sanitized content
  keeps `class`, so untrusted markup can *name* component classes — but the
  scope stamp only lands on template elements, so scoped rules never match
  injected nodes. Unscoped styles would. Pinned as the trade-off it is.

## Proposals (none implemented)

1. **Warn on top-level destructuring in setup scripts.** The scanner already
   knows it skipped one (`let`/`const` followed by `{` or `[` at depth 0);
   a `console.warn` naming the line would turn the silent trap into a
   guided one. Cost: a few lines in `transformSetupScript`. Risk: none —
   behavior unchanged.
2. **Document the sort()/reassign difference** in `docs/reactive-data.md` or
   `template-syntax.md` next to `:key`, and `:attrs`'s falsy semantics in
   `template-syntax.md`. Docs-only.
3. **Rejected: property-vs-attribute special-casing** (`value`, `checked`,
   `selected` as DOM properties in `:attrs`). It's what Vue does, but it
   changes `:attrs`'s one rule ("entries become attributes") into two, and
   the manual `@input` + assignment pattern already covers forms. Reconsider
   only if a `:model` directive is ever proposed, where it belongs.
4. **Rejected: blocking `//` protocol-relative URLs in the sanitizer.** They
   resolve under the page's scheme, which the allowlist already admits by
   name; blocking them would also block legitimate same-scheme links. The
   test pins the decision so it stays deliberate.

## Open

- Whether the destructuring warn (1) should also cover `let a = 1, b = 2`
  (the second name works today, but only by the new-key sweep).
- Whether `sanitizeHTML` should cap input depth explicitly instead of
  meeting RangeError somewhere above 2000 levels (it survives 500; the
  browser parser flattens around 512 anyway).

---

## 2026-07-15 — decisions (same day, user call)

Proposal 1 is **superseded, upward**: not a warn but real support —
top-level destructuring declarations become reactive scope assignments, and
the multi-declarator accident (`let a = 1, b = 2` registering only `a`) is
fixed in the same pass. Plan: `2026-07-15.setup-destructuring.md`. Both open
questions above close with it: the warn is moot, and multi-declarators are
covered by the fix rather than a warn.

The sanitizer depth question closes as: **cap at 512 with a named
RangeError** thrown to the caller. 512 matches what browser parsers tolerate
before flattening, so no legitimate document loses anything; the explicit
error replaces an incidental stack overflow with a message that tells the
template author what to fix.

Rejection 3 is **partially reversed, by redesign rather than reversal**: the
argument against special-casing `value`/`checked`/`selected` *inside
`:attrs`* stands, but the same capability lands as dedicated `:value` /
`:checked` / `:selected` directives that write the DOM property — `:attrs`
keeps its one rule. Plan: `2026-07-15.property-directives.md`.

Rejection 4 (protocol-relative URLs) stands. `:model` and a URL destination
policy stay deferred — to be looked at after these land.

The tests pinning the old behaviors (destructuring silence, `:attrs` value
detachment as the only option, sanitizer surviving-then-crashing) will be
updated to assert the new ones; that is the point of having pinned them
first.
