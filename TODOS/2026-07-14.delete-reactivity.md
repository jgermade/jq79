# `delete` on a store is invisible

**Status:** proposal. Nothing implemented.

## The problem

The store proxy traps `get` and `set`, but not `deleteProperty`. So
`delete data.user` removes the key from the raw object and tells nobody: no
`notify`, so effects that rendered `user` keep showing the deleted value; no
`$on`/`$onAny` call; and if the deleted value was a nested store, its bridge
stays subscribed — the holder keeps re-notifying changes of a store it no
longer holds.

It's not only the explicit `delete` statement. Array methods shrink through
[[Delete]] too: `arr.pop()` deletes the last index before writing `length`, so
today a pop is only *half* heard (the `length` write) and the removed slot's
own change is silent.

Found in the 2026-07-14 review, alongside the nested-component range bug
(`2026-07-14.nested-component-ranges.md`) — separate mechanism, so separate
plan.

## The proposal

A `deleteProperty` trap, symmetric with `set`:

```ts
deleteProperty(target, key) {
  if (typeof key !== "string") return Reflect.deleteProperty(target, key)
  const had = Object.prototype.hasOwnProperty.call(target, key)
  const deleted = Reflect.deleteProperty(target, key)
  if (deleted && had) {
    const dotKey = path ? `${path}.${key}` : key
    unbridge(dotKey)
    notify(dotKey, undefined)
  }
  return deleted
}
```

- **Notifies with `undefined`**, which is what a read of the key returns
  afterwards — a listener can't tell "deleted" from "set to undefined", and
  that's fine: for rendering they're the same fact.
- **Not a new-key sweep.** Effects that read the key while it existed tracked
  its dotKey (the `get` trap records the read whether or not the key is
  present), so plain dep matching wakes exactly them.
- **`unbridge` mirrors `set`'s behavior**: the exact path is unbridged, and
  only that. Bridges *under* a deleted subtree (a store sitting at `a.s` when
  `a` is deleted) are not chased — `set` doesn't chase them either when it
  overwrites `a`. If that asymmetry ever bites, it bites both traps and gets
  fixed in both.
- **Deleting a missing key notifies nobody** (`had` guards it), same as JS.

`with`-scoped setup scripts need no change: the scope proxy's `has` doesn't
trap deletes, and `delete` doesn't delegate up prototype chains, so a
scope-local delete can't reach the store by accident.

## What it costs

One trap, a few lines. Every `delete` now runs listener dispatch — the same
cost a `set` already pays, on an operation that's far rarer.

## Rejected

- **Documenting it as a limitation instead** ("use `data.user = undefined`").
  Cheaper, but the store's contract everywhere else is "mutate it like a plain
  object and the DOM follows" — `delete` is the one plain-object mutation that
  breaks the contract silently. A gap you have to document around is a bug.
- **Treating a delete as a new-key sweep** (re-run all effects): the sweep
  exists because an effect can't track a name that didn't exist when it read
  it. A deleted key *did* exist for whoever depended on it, so the targeted
  path is both sufficient and cheaper.

## Open

- Nothing known. `Object.keys`/`for…in` over a store aren't tracked today
  (no `ownKeys` trap) and that stays as it is — a template can't iterate an
  object's keys anyway (`:each` takes arrays), so nothing observable changes.

## 2026-07-14 — implemented

The trap landed as sketched, and the first template test caught what the
proposal missed: notifying is not enough. A template expression resolves its
names through `with ($scope)`, which asks the proxy's [[HasProperty]] — and a
deleted key isn't there anymore, so `user` fell through to `globalThis`, the
*whole* expression died of ReferenceError (caught, rendered as ""), and
`{{ user ? user.name : "none" }}` showed empty instead of its else branch.
Setup scripts never had this problem: their scope proxy claims every
non-global name regardless.

So each wrapped object keeps a **tombstone set**: keys deleted off it. The
new `has` trap claims a tombstoned key (resolving it to `undefined`, like a
key set to `undefined`), and a later `set` clears the claim. Lazily allocated
— a store nobody deletes from pays nothing.

The cost the proposal didn't list: `"user" in store` stays `true` after
`delete store.user`. That's the price of `with` resolution and it's confined
to reads of `in` against the proxy; `Object.keys` and spreads are honest.

Tests: `$on`/`$onAny` on delete, nested dot paths waking effects, `pop()`
notifying its dead slot, unbridging a deleted nested store, deleting a missing
key notifying nobody (tests/reactive.test.ts), and the template else-branch
case (tests/renderComponent.test.ts). `docs/reactive-data.md` now says deletes
are heard.
