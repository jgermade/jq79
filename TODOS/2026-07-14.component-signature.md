# Component signature: declared props, in both script modes

**Status:** proposal. Nothing implemented.

## The problem

A component has no signature. Props arrive from the parent, get seeded into the
store before the scripts run (`jq79.ts:194`) and re-synced by an effect
(`jq79.ts:206`), and the setup script sees them through `with`. Nothing ever
declares them, which means:

- **No defaults.** A prop with a fallback has to be faked as internal state
  (`let count = 0`), which lies about where the value comes from.
- **No API.** The component doesn't know its own prop names, so nothing can warn
  about a typo'd attribute, and no `.d.ts` can be emitted for it.
- **`:setup="{ user }"` is a lie.** The attribute's value is read by *nobody* —
  grep for it. It's a comment shaped like code, which is worse than no comment.

And the two script modes disagree about where props even live: a setup script
reads them by name, a factory script reads them off `$data` — its parameter is
the instance context, not the props.

## The proposal

**One concept — the component's prop signature — written in the place each mode
already puts its inputs.** Position is fixed, so the signature can be read
*statically*, straight from the component source, without executing anything.

```html
<!-- setup mode: the names live in the store, `with` resolves them -->
<script :setup="{ label = 'Total', step = 1 }">
  let count = 0
  const inc = () => { count += step }
</script>
```

```html
<!-- factory mode: props first, context second -->
<script>
export default ({ label = "Total", step = 1 }, { $data, $effect }) => {
  $data.count = 0
  const inc = () => { $data.count += step }
  return { label, inc }
}
</script>
```

The rule that makes it explainable in one line, and which the codebase already
follows without a single exception: **what carries a `$` comes from the library;
what doesn't comes from the parent.**

### What the signature does

1. **Declares** the prop names.
2. **Pre-declares** them on the store, so the template can bind to them even when
   the parent passes nothing (same mechanism already used for script `let`s at
   `jq79.ts:1113`).
3. **Seeds the defaults** into the store *before the first render*. This is the
   part that needs runtime support in factory mode too: JS itself applies
   `count = 0` when it invokes the factory, but that value only exists inside the
   function body — the template would still see `undefined`. So the runtime reads
   the pattern from the source string and seeds the store itself. One
   implementation, shared by both modes.

Reading the pattern is string-level work: names, defaults, comma-splitting that
respects strings and brackets. `skipString` and the depth counter in
`transform.ts` already do exactly that. No parser, no symbol table.

### The empty slot is part of the declaration

Position is fixed, so a factory that wants no props still has to leave the hole —
and the hole means something:

```js
export default (_,  { $effect }) => {}   // "I don't use props" — undeclared, permissive (today's behavior)
export default ({}, { $effect }) => {}   // "this component has no props" — closed signature
export default ({ user })        => {}   // only props; don't write a ctx you don't use
```

`{}` is a closed signature: the dev server can warn about any attribute the
component never declared. `_` declares nothing and keeps the current lenient
behavior. Same distinction in setup mode: `<script :setup>` declares nothing,
`<script :setup="{}">` declares zero props.

(ESLint's `no-unused-vars` defaults to `after-used`, so an unused `_` sitting
before a used parameter is not reported.)

## The one asymmetry, and why we keep it

In setup mode, a prop name is never a lexical binding — it's a store entry, and
`with` re-resolves it on every read. So it stays live, primitives included.

In factory mode, the pattern creates **real JS bindings**, and destructuring
copies. Which means:

```js
export default ({ cart, discount = 0 }, { $props, $effect }) => {
  cart.items.push(item)                        // ✅ live: you captured the proxy reference
  $effect(() => log(discount))                 // ❌ frozen: you captured a number
  $effect(() => log($props.discount))          // ✅ live: read through the object
}
```

Objects survive destructuring (you copied the reference, and every read of
`cart.x` still goes through the proxy). Only **primitives the parent reassigns**
go stale — the sync effect writes the store key, and your local copy never hears
about it.

So the ctx gains `$props`: the live view, under a different name. Two names, two
behaviors, nothing lying. The docs line is one sentence, and it's one a JS
programmer already believes: *destructuring copies.*

**Rejected: making the destructured names reactive.** It's what Svelte 5 and Vue
3.5 do, and it needs what they have — a compiler. `with` doesn't help: it only
intercepts names that *don't* resolve lexically, and a destructured `count` does.
To route it to the store you'd have to strip the pattern and rewrite every *read*
of `count` to `$data.count`, which needs real scope analysis (shadowing, shorthand
`{ count }`, property keys, nested functions). Shipping a JS parser to the browser
is the one thing jq79 says it doesn't do. And running the factory body under
`with` would also mean dropping `"use strict"` (`with` is a SyntaxError in strict
mode) from code whose entire selling point is that it's a standard, strict ES
module.

The IDE, notably, is *not* an argument either way: it reads the `.html` source as
authored and never sees a runtime transform. What breaks isn't highlighting, it's
semantic fidelity — and debugging, since a rewritten factory body would start
reporting shifted line numbers with no source map, a tax factory scripts don't pay
today.

## What it buys, once the signature is static

- **Dev server:** warn on an attribute no component declares; warn on a required
  prop the parent never passes.
- **Vite plugin:** emit a `.d.ts` per component, so `<UserCard :user="x" />` in
  another `.html` is actually type-checked.
- **Runtime:** defaults reach the template, without invoking the factory.
- **Docs:** a component's API stops being a comment.

None of it works if the signature can move, or if you have to run the factory to
find out what it takes. `_` isn't noise — it's what keeps the hole where the
tooling knows to look.

## Breaking change: factory parameter order

Today the factory's *first* parameter is the ctx (`({ $data, $effect })`). Under
this proposal it's the props. Every existing factory breaks, and it breaks
*silently*: `$data` would come back `undefined`.

Arity can't disambiguate — under the new contract a single parameter is a
legitimate prop signature (`({ user })`), and under the old one it was the ctx.
Both shapes are valid and look identical.

The fix, since the runtime has the source as a string: when parsing the first
parameter's pattern, if it contains `$`-prefixed names (`$data`, `$effect`,
`$emit`, …), throw an explicit migration error — *"the factory signature is
`(props, ctx)`; use `_` if you don't need props"*. Not a heuristic that changes
behavior, just a message: deterministic, impossible to confuse with the new form
(props never carry a `$`, by the same convention the whole codebase follows), and
removable in a later version.

We're on 0.3.x. If this is going to happen, now is when it's cheap.

## Open decisions

- **Explicit `undefined`.** Apply the default when the value *is* `undefined`, or
  when the key is *absent*? Recommend `undefined` — it's what Vue and Svelte do,
  it sidesteps the `in` edge created by pre-declaration, and it's what people
  expect. The cost: a parent can't force `undefined` over a default.
- **Re-applying a default.** If the parent later sets the prop to `undefined`, do
  we re-apply? Recommend no: apply once, at setup. Doing it continuously needs an
  effect that reads and writes the same variable — the self-waking loop
  `docs/development.md` warns about.
- **Undeclared props.** Opt-in, per the `_` vs `{}` distinction above. A component
  with no signature behaves exactly as it does today.

## Implementation sketch

- `transform.ts`: parse a destructuring pattern from a string → `{ name, default }[]`.
  Reuse `skipString` + depth counting. One function, used by both modes.
- `transform.ts`: read the `:setup` attribute value with it; read the factory's
  first parameter with it (and detect the legacy `$`-pattern to raise the
  migration error).
- `jq79.ts`: pre-declare + seed defaults on the store before the scripts run,
  next to the existing pre-declaration at `jq79.ts:1113`.
- `jq79.ts`: pass props as the factory's first argument, ctx as the second; add
  `$props` (the live store view) to the ctx.
- Tests, `docs/setup-scripts.md`, `docs/components.md`, and the tutorial's
  exercises that use props.
