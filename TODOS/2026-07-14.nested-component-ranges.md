# Nested components in `:each`/`:if`: fragments have no identity

**Status:** proposal. Nothing implemented.

## The problem

`renderNestedComponent` returns a `DocumentFragment` holding an anchor comment;
the component's markers and DOM are inserted next to that anchor once the
definition resolves. A `DocumentFragment` *empties itself* into the parent on
insertion — after the first `insertBefore` its identity is worthless: it has no
children, no parent, and `fragment.nextSibling` is always `null`.

Two callers keep that fragment as their handle to the rendered chunk:

- **`renderEach`** stores it as `entry.node` and uses it to diff positions
  (`prevNode.nextSibling !== entry.node`) and to move entries
  (`insertBefore(entry.node, …)`). For component entries both are no-ops on an
  empty fragment, so **a keyed list of components never reorders** — verified:
  `<Card :each="item in items" :key="item.id">` with the list reordered to
  `[3, 1, 2]` keeps rendering `[1, 2, 3]`. Worse, `prevNode = entry.node` makes
  `prevNode.nextSibling` `null`, so in a mixed list the *following* siblings get
  appended at the end of the container — silent scrambling.
- **`renderConditional`** stores it as `current` and removes it with
  `current.parentNode?.removeChild(current)` — a no-op. The instance's
  `destroy()` (via `onDispose`) does remove the component's markers and content,
  but the anchor comment `renderNestedComponent` created is *outside* those
  markers and nobody removes it: **toggling `:if` on a component leaks one
  comment node per cycle** — verified, 5 toggles add 5 comments. The same
  applies to removing component items from a `:each`.

The test suite doesn't catch any of this because no test combines `:each` or
branch-switching `:if` with a component tag. Plain elements are fine: for them
`renderNode` returns the element itself, a perfectly stable handle.

## The proposal

Give a nested component's output **stable boundaries**, and make the two
callers track *ranges* instead of node identities.

1. **`renderNestedComponent` brackets its output with two anchors** — the
   existing start comment plus an end comment — and inserts each instance's DOM
   *before the end anchor* instead of after the start one. Everything the
   component ever puts in the DOM now lives strictly between two nodes that
   never move on their own. (The end anchor is not optional: the content between
   the anchors is dynamic — the definition can resolve late, be swapped, be
   hot-replaced — so a "last child" captured at creation time can be detached
   by the time it's needed.)

2. **`renderEach` and `renderConditional` capture `{ first, last }` bounds**
   when they render a chunk, *before* inserting it: for an element, the element
   twice; for a fragment, its first and last children (the anchors). Moving an
   entry moves `[first..last]` inclusive, node by node; removing one removes the
   same range; the position diff compares against `entry.first` and walks on
   from `entry.last`. Removal by range is also what fixes the comment leak: the
   anchors are inside the range, so they leave with it.

3. **The late-upgrade path** (`HTMLUnknownElement`/dashed tag whose component
   key arrives later) replaces the placeholder with the same bracketed fragment
   and registers an `onDispose` that removes the range — so a disposed subtree
   doesn't strand the anchors when the placeholder (the only node its caller
   knows about) is already detached.

No public API changes; `hotReplace` is unaffected (it re-attaches against the
*component's own* markers, which sit between the anchors either way).

## What it costs

- One more comment node per nested-component usage site. Comments render
  nothing and the runtime already leans on them (`jq79`/`/jq79` markers, the
  `:if`/`:each` anchors).
- Moving a range is one `insertBefore` per node in it instead of one per
  fragment. Reordering N components touches each component's handful of
  top-level nodes — the same order of work the browser was already doing when
  the fragment moved its children.

## Rejected

- **Wrapping each nested component in a real element** (`<div style="display:
  contents">`, a custom element): a stable handle for free, but it changes the
  DOM shape users style against (child selectors, `:nth-child`, grids — the
  tutorial's own layout notes that a component's markers are "no grid items of
  their own"). Not worth it to fix bookkeeping.
- **Bracketing every `:each` entry with its own marker pair** instead of
  touching `renderNestedComponent`: fixes the list but not `:if`, adds two
  comments per item to *every* list (the common, plain-element case pays for
  the rare one), and leaves the anchor leak in place.
- **Returning a range type from `renderNode` everywhere**: honest, but it
  forces every caller — including the ones a stable `Node` serves fine — to
  speak ranges. The two call sites that reposition/remove are the only ones
  that need them.

## Open

- The late-upgrade path still can't *reposition*: if the placeholder was itself
  the root of a `:each` item (a factory script merging the component key after
  `await`), the entry's handle is the detached placeholder, and reordering that
  list re-inserts the placeholder instead of moving the component. Today's
  behavior is equally broken and additionally leaks; the full fix needs the
  entry to re-capture its bounds on upgrade, which needs a channel from the
  upgrade effect back to the entry — left out until someone actually hits it.
- `delete` on a store is still non-reactive (no `deleteProperty` trap) — found
  in the same review, unrelated mechanism, deserves its own plan.

## 2026-07-14 — implemented

Shipped as proposed, the same day. What landed in `src/jq79.ts`:

- `NodeRange` (`{ first, last }`) with three helpers: `boundsOf` (an element is
  its own range; a fragment's range is its first/last children), `removeRange`
  and `moveRangeAfter` — both walk `[first..last]` inclusive by sibling, so
  whatever the component put between its anchors travels with them.
- `renderNestedComponent` brackets its output with a second anchor
  (`<!--/Key-->`) and inserts each instance's DOM before it.
- `renderConditional` keeps the active branch as a `NodeRange`; `renderEach`
  keeps one per entry (`EachEntry.node` → `EachEntry.range`), and the position
  diff compares `entry.range.first` / walks on from `entry.range.last`.
- The late-upgrade path registers an `onDispose` that removes the range, since
  its caller only knows the placeholder the swap detached.

Four tests in `tests/Component79.test.ts` under "nested components": keyed
reorder preserving instances (plus a trailing sibling that must not move),
`:each` removal churn, `:if` toggle churn, and the upgraded-late-component
branch teardown. Each reproduced the bug before the fix.

The two open items stay open: repositioning an upgraded placeholder at the
root of a `:each` entry, and the missing `deleteProperty` trap.
