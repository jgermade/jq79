# `:model` ‚Äî two-way binding on component tags

**Status:** decided (design converged in conversation; the user's sketch set
the grammar, the forks below were the user's call). Not yet implemented.

## The problem

A parent that hands state to a child component and wants the child's edits
back has to wire both directions by hand: a `:prop` down, plus an emit the
parent catches on a *wrapping element* (component tags don't hold listeners ‚Äî
they render as two comment anchors). For form-shaped children (a login form,
a field component) that's four attributes of ceremony per field, and the
wiring is the same every time: prop down, event up, assign back.

`2026-07-15.property-directives.md` deferred `:model` when `:value` /
`:checked` / `:selected` shipped, noting it would build on them. This is that
proposal ‚Äî for **component tags**; the native-element form stays parked
(below).

## The proposal

One directive, name optionally carried by the modifier (the second
value-carrying modifier, after `:html.allowed`):

```html
<LoginForm :model.uname="uname" :model.password="password" />
<EmailField :model="email" />
```

Each `:model[.name]="expr"` desugars to **prop down + writeback up**:

- **Prop down**: the expression is passed as a prop named after the model ‚Äî
  the modifier name (kebab‚Üícamel, since HTML lowercases attribute names), or
  `model` for the modifier-less form (`default` is a reserved word; an
  expression like `:value="default"` in the child could never read it). It
  rides the existing prop machinery: seeded at creation, kept live by the
  same sync effect as any `:prop`. The child may ignore it, but a parent
  reset (`email = ""`) reaches the child's input ‚Äî without the prop this
  isn't two-way, it's upward collection, and initial values never land.
- **Writeback up**: the child emits **one event, one contract**:

  ```js
  $emit("model:update", { name: "uname", value })   // named model
  $emit("model:update", { value })                  // name omitted ‚Üí "default"
  ```

  The parent wires `instance.on("model:update", ...)` at the usage site and
  assigns `expr = value` for the matching model. The expression must be an
  assignment target (`uname`, `user.name`); it's evaluated in the parent
  scope, so it writes through the store (reactively) and through `:with`
  narrowing (its set trap).

- Shorthand, same rule as props: `:model.uname` ‚â° `:model.uname="uname"`,
  `:model` ‚â° `:model="model"`.

## Semantics pinned

- **Name normalization**: the modifier arrives lowercased from the HTML
  parser, so it's declared kebab-case and normalized with `kebabToCamel` ‚Äî
  and the payload's `name` gets the same normalization, so
  `:model.user-name` matches `{ name: "userName" }` (or `"user-name"`).
  A payload `name` of `undefined` means `"default"`, which is what the bare
  `:model` binds; an explicit `{ name: "default" }` matches it too.
- **The echo terminates**: parent assigns ‚Üí the prop sync effect rewrites the
  child's store key with the same primitive ‚Üí skipped by the store's
  same-value guard (`reactive.ts`), and `:value` in the child already skips
  same-string writes (the caret guard). The loop the wiring *looks* like it
  creates never runs.
- **Fail loud, do nothing** (an event contract, so "loud" beats "closed"):
  - a payload that isn't an object ‚Üí `console.warn`, ignored;
  - a `name` no `:model[.name]` on the tag binds ‚Üí `console.warn` listing
    the bound names (typos must not be silent);
  - an expression that isn't assignable (`:model.x="a + b"` won't compile as
    `a + b = $value`) ‚Üí `console.warn` **at wiring time**, not first emit;
  - `payload.value` absent ‚Üí assigns `undefined` (one rule, no special case).
- **`$value` in the assignment**: the payload value is passed as the `$value`
  parameter (outside the `with`, like `$event` in handlers), so a store key
  literally named `$value` would shadow it ‚Äî same caveat `$event` already
  has, same answer: don't do that.
- **Collisions**: `:model.uname` and an explicit `:uname` on the same tag
  bind the same prop ‚Äî the model's binding wins, with a `console.warn`.
- **On a plain element** (`<input :model="x">`): skipped with a
  `console.warn` ‚Äî components only in this pass. The warn is suppressed on
  unknown/dashed tags (the may-become-a-component-later path), which upgrade
  into the full wiring when the key arrives.
- The writeback listener uses `instance.on()`, not a DOM listener on a
  marker: it survives hot-replace (markers are per render generation) and
  works detached. It's wired per instance, inside the same effect that
  creates it.

## Parked: `:model` on native elements

`<input :model="name">` as sugar for `:value="name"` +
`@input="name = $event.target.value"` (and `:checked`/`@change` on
checkboxes, `:value`/`@change` on selects). Builds on shipped pieces, but
carries its own decisions (radio groups, multi-selects, IME composition) and
none of this proposal depends on it. Same directive name, disjoint target ‚Äî
adding it later breaks nothing decided here.

## What it costs

- A models branch next to the props collector in `renderNestedComponent`
  (~10 lines), the writeback wiring in the instance-creation effect (~25
  lines), the plain-element warn in `renderNode`, docs, tests.
- The second value-carrying modifier. Mechanically free (another attribute
  pattern), conceptually already paid for by `:html.allowed`.

## Rejected

- **Writeback only, no prop down** ‚Äî user's call against: without the prop
  the binding is one-directional collection; parent resets and initial
  values silently never reach the child.
- **Per-name events** (`update:uname`, Vue-style) ‚Äî one event name with the
  name in the payload keeps the child's emit site to one contract and the
  routing trivial; it's also the user's sketched grammar.
- **Non-object payload as default-model shorthand**
  (`$emit("model:update", value)`) ‚Äî ambiguous the moment the value *is* an
  object with `name`/`value` keys. Strict `{ name?, value }`, warn on the
  rest.
- **Native elements in this pass** ‚Äî user's call; parked above, as the
  property-directives plan anticipated.
- **DOM listener on the instance's start marker** for the writeback ‚Äî dies
  on hot-replace (new markers per generation) and needs a private field;
  `emitListeners` is the channel built to survive exactly that.
- **`default` as the default prop name** ‚Äî reserved word; unreadable from
  child expressions.

## Open

- Nothing known.

## 2026-07-15 ‚Äî addendum: `$emit` in template expressions

Implementing the child side of the sketch surfaced a hole the proposal had
missed: `$emit` reaches setup scripts as a function *parameter* (outside the
`with`), so a template-inline handler ‚Äî the sketch's
`@input="$emit('model:update', { value: ... })"` ‚Äî resolved `$emit` to
`undefined`, silently. Every emit in the repo routed through a setup
function (the Stepper pattern) precisely because of this.

Fixed at the template's root scope: `renderWith` hands `renderNodes` a proxy
over the store that answers `$emit` via has/get (unless the store shadows
the name) ‚Äî the same spirit as the setup scripts' `scriptScope` has-trap.
Never an own key, so `Object.keys`, snapshot spreads and the component-key
scan don't see it, and all other reads forward through the reactive store,
keeping tracking intact. Only `$emit` is exposed ‚Äî `$self`/`$mounted` make
no sense mid-template; the seam is there if that ever changes.

## 2026-07-15 ‚Äî addendum: writebacks run untracked (and the hazard that wasn't)

Extending the test suite raised a question: the writeback can run while an
effect is mid-run (a child emitting from its setup script runs inside the
parent's *creation* effect; a `$:` that emits runs inside itself), and the
reads a path assignment makes (`user` in `user.name = $value`) land in
whatever dep set is on top of the tracker stack. Theorized consequences:
the parent re-creating the child per write, or a read-write handler waking
its own emitter in a loop.

Tested with the wrapping removed: **neither happens.** Two existing
containments, independently sufficient ‚Äî dependencies are per store, so a
read of the *parent's* store donated to a *child's* effect is never
notified (cross-store deps are inert); and the creation effect's
`nextDef === currentDef` guard makes a spurious same-store wake a no-op
that self-heals on its next pass (the re-run re-tracks only what it reads).

The `untracked()` wrapping stays anyway, on both the writeback and the tag
handlers (see the events plan) ‚Äî as semantics, not as a bug fix: an element
handler never runs inside an effect, and a tag handler should behave
identically no matter when the emit fires. "What a handler reads is
nobody's dependency" shouldn't hinge on the two containments staying as
they are. The suite pins the visible behavior (setup-emit initialization,
`$:`-driven emits settling in one pass) either way.

## 2026-07-15 ‚Äî addendum: the flaky-points audit

Asked "what's still weak here", three candidates; one survived contact with
the evidence.

**Ghost writebacks after teardown ‚Äî already contained.** A closure the old
child leaked (a timer emitting after its usage site swapped it out) cannot
keep writing the parent: `destroy()` nulls `startMarker`, so `$emit`'s
stale-generation guard (`marker === this.startMarker`) skips the `on()`
channel ‚Äî which is where the writeback and the tag handlers live ‚Äî and the
DOM dispatch leaves a detached comment for nowhere. Pinned by test.

**Trailing line comments ‚Äî a real silent hole, wider than `:model`.** The
writeback compiled `${expr} = $value` on one line, so
`:model="uname // the username"` put the assignment *inside the comment*:
it compiled cleanly (no wiring warn) and dropped every update without a
word ‚Äî exactly the failure class this plan promised not to have. And the
general case was already broken before `:model` existed: `compileExpr`
wraps everything in a single-line `return (expr);`, so a trailing comment
in *any* template expression swallowed the `);` and the expression never
compiled (silently, as always). Fixed at both layers with a newline ‚Äî
`return (expr\n);` in `compileExpr` (ASI doesn't apply inside parens, so
nothing else changes) and `${expr}\n= $value` in the writeback, which
makes trailing comments simply *work* rather than warn.

**Warn floods ‚Äî cosmetic, fixed.** The off-contract warns (unknown name,
bad payload) fired per emit: an input with a typo'd name meant one warn per
keystroke. Now once per mistake per instance.

Also for the record: the new tests are all synchronous ‚Äî no timers, no
waitFor, nothing for CI to be flaky *with*.

## 2026-07-15 ‚Äî tutorial lesson

`tutorial/03-components/05-two-way-binding/` teaches the collapse: the
starting `NameField.html` copies its `initial` prop into a local `value`
once at setup (the shortcut people reach for before `:model`), so neither
direction works afterward ‚Äî typing never reaches the parent, and the
parent's own reset never reaches the field, since `value` stops looking at
the prop after the first render. The solution deletes the copy, binds the
input straight to `model`, and adds one `$emit('model:update', ‚Ä¶)` line;
both symptoms disappear. Pinned in `tests/tutorial.test.ts`, added to the
staged-failure list next to keys-and-identity and untrusted-html.

Building it surfaced a real, general timing point worth recording: effect
notification in `reactive.ts` runs synchronously inside the write that
triggers it (no microtask batching), so in a setup script, a store write
can synchronously wake an effect that reads a *later* line's variable
before that line has executed. Concretely, `const NameField = await
import(...)` followed by `let name = "Ada"` renders the component tag's
prop seed with `name` still undefined - the write to `NameField` (a new
key) synchronously re-runs the tag's creation effect before the next
statement assigns `name`. The exercise's `app.html` declares `name` (and
`reset`) *before* the `await import(...)` line to sidestep it. Not a `:model`
bug - the same ordering would bite any prop passed to a component awaited
after the prop's own declaration - and not introduced by this change, just
run into while writing the lesson. Left as a documentation opportunity
rather than expanded here.
