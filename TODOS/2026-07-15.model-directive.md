# `:model` — two-way binding on component tags

**Status:** decided (design converged in conversation; the user's sketch set
the grammar, the forks below were the user's call). Not yet implemented.

## The problem

A parent that hands state to a child component and wants the child's edits
back has to wire both directions by hand: a `:prop` down, plus an emit the
parent catches on a *wrapping element* (component tags don't hold listeners —
they render as two comment anchors). For form-shaped children (a login form,
a field component) that's four attributes of ceremony per field, and the
wiring is the same every time: prop down, event up, assign back.

`2026-07-15.property-directives.md` deferred `:model` when `:value` /
`:checked` / `:selected` shipped, noting it would build on them. This is that
proposal — for **component tags**; the native-element form stays parked
(below).

## The proposal

One directive, name optionally carried by the modifier (the second
value-carrying modifier, after `:html.allowed`):

```html
<LoginForm :model.uname="uname" :model.password="password" />
<EmailField :model="email" />
```

Each `:model[.name]="expr"` desugars to **prop down + writeback up**:

- **Prop down**: the expression is passed as a prop named after the model —
  the modifier name (kebab→camel, since HTML lowercases attribute names), or
  `model` for the modifier-less form (`default` is a reserved word; an
  expression like `:value="default"` in the child could never read it). It
  rides the existing prop machinery: seeded at creation, kept live by the
  same sync effect as any `:prop`. The child may ignore it, but a parent
  reset (`email = ""`) reaches the child's input — without the prop this
  isn't two-way, it's upward collection, and initial values never land.
- **Writeback up**: the child emits **one event, one contract**:

  ```js
  $emit("model:update", { name: "uname", value })   // named model
  $emit("model:update", { value })                  // name omitted → "default"
  ```

  The parent wires `instance.on("model:update", ...)` at the usage site and
  assigns `expr = value` for the matching model. The expression must be an
  assignment target (`uname`, `user.name`); it's evaluated in the parent
  scope, so it writes through the store (reactively) and through `:with`
  narrowing (its set trap).

- Shorthand, same rule as props: `:model.uname` ≡ `:model.uname="uname"`,
  `:model` ≡ `:model="model"`.

## Semantics pinned

- **Name normalization**: the modifier arrives lowercased from the HTML
  parser, so it's declared kebab-case and normalized with `kebabToCamel` —
  and the payload's `name` gets the same normalization, so
  `:model.user-name` matches `{ name: "userName" }` (or `"user-name"`).
  A payload `name` of `undefined` means `"default"`, which is what the bare
  `:model` binds; an explicit `{ name: "default" }` matches it too.
- **The echo terminates**: parent assigns → the prop sync effect rewrites the
  child's store key with the same primitive → skipped by the store's
  same-value guard (`reactive.ts`), and `:value` in the child already skips
  same-string writes (the caret guard). The loop the wiring *looks* like it
  creates never runs.
- **Fail loud, do nothing** (an event contract, so "loud" beats "closed"):
  - a payload that isn't an object → `console.warn`, ignored;
  - a `name` no `:model[.name]` on the tag binds → `console.warn` listing
    the bound names (typos must not be silent);
  - an expression that isn't assignable (`:model.x="a + b"` won't compile as
    `a + b = $value`) → `console.warn` **at wiring time**, not first emit;
  - `payload.value` absent → assigns `undefined` (one rule, no special case).
- **`$value` in the assignment**: the payload value is passed as the `$value`
  parameter (outside the `with`, like `$event` in handlers), so a store key
  literally named `$value` would shadow it — same caveat `$event` already
  has, same answer: don't do that.
- **Collisions**: `:model.uname` and an explicit `:uname` on the same tag
  bind the same prop — the model's binding wins, with a `console.warn`.
- **On a plain element** (`<input :model="x">`): skipped with a
  `console.warn` — components only in this pass. The warn is suppressed on
  unknown/dashed tags (the may-become-a-component-later path), which upgrade
  into the full wiring when the key arrives.
- The writeback listener uses `instance.on()`, not a DOM listener on a
  marker: it survives hot-replace (markers are per render generation) and
  works detached. It's wired per instance, inside the same effect that
  creates it.

## Parked: `:model` on native elements

`<input :model="name">` as sugar for `:value="name"` +
`@input="name = $event.target.value"` (and `:checked`/`@change` on
checkboxes, `:value`/`@change` on selects). Builds on shipped pieces, but
carries its own decisions (radio groups, multi-selects, IME composition) and
none of this proposal depends on it. Same directive name, disjoint target —
adding it later breaks nothing decided here.

## What it costs

- A models branch next to the props collector in `renderNestedComponent`
  (~10 lines), the writeback wiring in the instance-creation effect (~25
  lines), the plain-element warn in `renderNode`, docs, tests.
- The second value-carrying modifier. Mechanically free (another attribute
  pattern), conceptually already paid for by `:html.allowed`.

## Rejected

- **Writeback only, no prop down** — user's call against: without the prop
  the binding is one-directional collection; parent resets and initial
  values silently never reach the child.
- **Per-name events** (`update:uname`, Vue-style) — one event name with the
  name in the payload keeps the child's emit site to one contract and the
  routing trivial; it's also the user's sketched grammar.
- **Non-object payload as default-model shorthand**
  (`$emit("model:update", value)`) — ambiguous the moment the value *is* an
  object with `name`/`value` keys. Strict `{ name?, value }`, warn on the
  rest.
- **Native elements in this pass** — user's call; parked above, as the
  property-directives plan anticipated.
- **DOM listener on the instance's start marker** for the writeback — dies
  on hot-replace (new markers per generation) and needs a private field;
  `emitListeners` is the channel built to survive exactly that.
- **`default` as the default prop name** — reserved word; unreadable from
  child expressions.

## 2026-07-15 — addendum: `$emit` in template expressions

Implementing the child side of the sketch surfaced a hole the proposal had
missed: `$emit` reaches setup scripts as a function *parameter* (outside the
`with`), so a template-inline handler — the sketch's
`@input="$emit('model:update', { value: ... })"` — resolved `$emit` to
`undefined`, silently. Every emit in the repo routed through a setup
function (the Stepper pattern) precisely because of this.

Fixed at the template's root scope: `renderWith` hands `renderNodes` a proxy
over the store that answers `$emit` via has/get (unless the store shadows
the name) — the same spirit as the setup scripts' `scriptScope` has-trap.
Never an own key, so `Object.keys`, snapshot spreads and the component-key
scan don't see it, and all other reads forward through the reactive store,
keeping tracking intact. Only `$emit` is exposed — `$self`/`$mounted` make
no sense mid-template; the seam is there if that ever changes.
