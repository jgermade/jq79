# `:html.allowed` — a per-element destination policy for sanitized content

**Status:** decided (design converged in conversation, user's call on the
grammar). Not yet implemented.

## The problem

`sanitizeHTML` asks one question of a URL: *how* it is reached. The protocol
check kills execution (`javascript:`, `data:`), which is the actual XSS
defense — but it never asks *where* the URL points. Sanitized user content
can link to a phishing domain or auto-load a tracking pixel (`<img src>`
leaks every reader's IP without a click) through URLs whose protocol is
impeccable.

Page-wide, that control belongs to CSP — the browser enforces it over the
whole DOM, including whatever never passed through `sanitizeHTML`. What CSP
cannot do is distinguish *zones of trust within one page*: a comments area
that allows nothing external next to a partner-content area that allows the
partner's CDN. That per-element decision can only live where the `:html`
lives.

## The proposal

Three layers, each useful alone:

1. **The primitive** — `sanitizeHTML(html, { allowUrl })`, where
   `allowUrl: (url: URL, tag, attr) => boolean` is consulted for `href`/`src`
   *on top of* (never instead of) the protocol check. The URL is resolved
   against `document.baseURI`, so relative URLs are judged as the
   same-origin destinations they really are. A throwing predicate rejects.
2. **The pattern sugar** — `allowedHosts(patterns)` compiles a
   comma-separated string or an array of host patterns into a predicate.
3. **The template surface** — `:html.allowed="expr"` on the `:html` element;
   the expression is evaluated in scope (reactively, inside the same effect)
   and normalized: string/array → `allowedHosts`, function → used as the
   predicate, anything else → deny all.

```html
<div :html="body" :html.allowed="'*.germade.dev'"></div>
<div :html="body" :html.allowed="['*.germade.dev', '*.germade.es']"></div>
<div :html="body" :html.allowed="url => url.hostname.endsWith('.germade.dev')"></div>
<div :html="body" :html.allowed="allowedFromStore"></div>
```

The value is always an **expression**, like every other `:` attribute — a
bare pattern literal (`:html.allowed="*.germade.dev"`) was considered and
rejected below.

## The pattern grammar

`host[:port]`, where:

- `*` matches **exactly one DNS label** — the TLS-certificate wildcard rule,
  deliberately *not* CSP's any-depth `*.`: `*.germade.dev` matches
  `a.germade.dev`, but neither `germade.dev` (write both to get the apex)
  nor `a.b.germade.dev` (that's `*.*.germade.dev`). One label is precise and
  predictable; any-depth is a surprise multiplier.
- No port means any port (`*.germade.dev` ≡ `*.germade.dev:*`); an explicit
  port must equal the URL's *effective* port (the explicit one, or the
  scheme's default — `germade.dev:443` matches `https://germade.dev/`).
- **Fail closed everywhere**: an invalid pattern matches nothing, a policy
  expression that evaluates to `undefined` denies every destination, a
  throwing predicate is a `false`. The attribute declares the intent to
  restrict; a broken policy must not quietly un-restrict.
- Under a pattern list, a URL with no host (`mailto:`) is rejected — hosts
  are what patterns speak. The function form can allow it.

## Defaults

`:html` without `.allowed` keeps today's behavior: protocol check only, any
destination. Restricting by default would silently break every existing
`:html` that renders markdown with external links or images, and the
destination is product policy, not a vulnerability — policies are declared,
not imposed. The composition story: **CSP as the page-wide floor the browser
enforces, `.allowed` as per-zone tightening declared in the template** — the
stricter one always wins.

## What it costs

- A `depth`-style parameter threaded through the sanitizer (an options
  object this time), the ~20-line pattern compiler, and one normalize + wire
  block in `renderNode`. `":html.allowed"` joins `CONTROL_ATTRS`.
- A new concept: the first directive modifier that carries its own value
  (event modifiers are flags). Mechanically free — it's just another
  attribute name to the parser.

## Rejected

- **Bare pattern literals** (`:html.allowed="*.germade.dev"`): would need
  try-evaluate-then-fall-back magic, whose failure mode is a typo'd function
  silently becoming a never-matching pattern. Expressions everywhere is the
  library's one rule for `:` attributes; the quoted string costs two
  characters.
- **Global configuration** ("restrict every `:html` in the app"): against
  the zero-config grain, and the page-wide job is CSP's. Explicit non-goal
  until a real user shows up with a hosting that can't set headers.
- **Extending to `:attrs` URLs in this pass**: `:attrs` values come from the
  component author, not from untrusted content — different trust level, no
  sanitizer involved.
- **CSP's any-depth `*.` semantics**: see the grammar — user's call, TLS
  rule wins.
- **A host allowlist built into the library** (predefined safe hosts):
  config that ages badly and false coverage — only `:html` content would
  honor it.

## 2026-07-15 — implemented

As proposed, the same day. `dom.ts` grew the `allowUrl` option on
`sanitizeHTML` (consulted for `href`/`src` after — never instead of — the
protocol check, URL resolved against `document.baseURI`, throw = no) and the
exported `allowedHosts` compiler; `jq79.ts` wires `":html.allowed"` (now in
`CONTROL_ATTRS`) through `normalizeAllowUrl` inside the same effect as
`:html`, so a store-held policy is reactive for free, and warns on an
orphaned `.allowed`.

Tests pin the grammar exactly as decided: single-label `*` (TLS rule —
`a.germade.dev` yes, apex and `a.b.` no), any-port default, effective-port
matching, comma-string/array equivalence, invalid-pattern and
undefined-policy and throwing-predicate all failing closed, `javascript:`
surviving an allow-all predicate, relative URLs judged same-origin, the
predicate's `(url, tag, attr)` contract, and the directive's reactivity.
Documented in `docs/template-syntax.md` under `:html`, with the CSP
composition note (page-wide floor there, per-zone tightening here).

## 2026-07-15 — tutorial lesson

`tutorial/01-basics/08-untrusted-html/` teaches the feature the staged-failure
way (like keys-and-identity): the starting file already renders hostile
comments through `:html`, so the sanitizer's protocol work is visible for
free, and the phishing link + tracking pixel that survive it are the failure
the solution fixes by adding `:html.allowed`. The preview prints each live
link's destination via `a[href]::after` and paints the pixel red, so the
policy's effect is watchable. Both halves are pinned in
`tests/tutorial.test.ts` — the *starting* file's claims too, since a
restrict-by-default change would gut the lesson without breaking its solution.
