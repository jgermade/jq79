# Destructuring declarations in setup scripts

**Status:** decided (see the decisions appendix in
`2026-07-15.weak-points-audit.md`). Not yet implemented.

## The problem

`let { a, b } = obj` at a setup script's top level keeps its keyword — the
scanner only rewrites `let <identifier>` — so its bindings are locals:
invisible to the template, not reactive, and silent. Svelte (whose script
style this borrows) supports it, so it's the first thing a migrating user
writes. The audit pinned the silence with tests; the decision is to support
it rather than warn about it.

A second accident rides along: `let a = 1, b = 2` rewrites only `a` — `b`
reaches the store as a bare assignment via the new-key sweep, working by
luck. Fixing declarators properly fixes both.

## The proposal

Rewrite a whole top-level declaration statement at once, instead of just
stripping the keyword and letting the main loop scan on:

- Match `let|var|const` followed by an identifier, `{` or `[` (today:
  identifier only), at depth 0 and statement start.
- Find the statement's end with `findStatementEnd`, split the declarator
  list at top-level commas keeping each segment's raw text (layout,
  comments and strings preserved), and re-scan each segment recursively so
  an `import(...)` inside an initializer is still rewritten.
- Identifier declarators emit as today (`count = 0`). Pattern declarators
  are parenthesized — `;({ a, b } = obj)` — because inside `with`, an
  assignment pattern writes every binding through the store proxy: that is
  the whole feature. The `;` guards against the previous line gluing to the
  `(`. The closing paren lands after the segment's last meaningful token,
  so a trailing line comment can't swallow it.
- `vars` collects the *binding* names, via a small recursive pattern reader:
  `{ a: x }` binds `x`, `{ a: { b } }` binds `b`, `[x, ...rest]` binds `x`
  and `rest`, defaults are stripped. (`parsePropsPattern` is the wrong tool
  here: it answers "which props", i.e. the keys — this needs the bindings.)
- `findStatementEnd` learns one more ASI fact it was missing: a line whose
  last meaningful character is `,` or `=` cannot end a statement, so a
  multi-line declarator list stays whole. This also makes `$:` statements
  slightly more correct, in the same spirit as the existing leading-token
  check.

## What it costs

- The declaration branch of `transformSetupScript` grows from 8 lines to a
  segment-splitting rewrite (~50 lines with the pattern reader), all built
  on the existing skip helpers.
- `const`-ness of a rewritten declaration was already lost for identifiers;
  patterns are no different.

## Rejected

- **A console.warn instead.** Was the original proposal; superseded by the
  user's call to support it outright. The warn would have been cheap, but
  it guides the user to a workaround for something the runtime can just do.
- **Full JS parsing.** The scanner stays a scanner: on malformed input it
  terminates and hands the broken source to the engine, which reports the
  real SyntaxError.

## Open

- A trailing line comment *between* a declarator's comma and the newline
  still cuts the statement early (walking backwards over a line comment
  needs forward context the scanner doesn't keep). The cut degrades exactly
  as today — the next declarator becomes a bare assignment caught by the
  new-key sweep — so it stays a known rough edge, not a blocker.

## 2026-07-15 — implemented

As proposed, the same day. `DECLARATION_START_RE` accepts identifier, `{`
and `[` targets (space optional before a pattern, so `const{a}` works);
`rewriteDeclarators` splits the list at top-level commas keeping raw
segments, re-scans each initializer (the `import()` rewrite reaches them),
parenthesizes pattern segments with the close placed before any trailing
comment, and `;`-guards a leading `(`. `patternBindings` reads the bound
names recursively; `patternCloseIndex` was extracted from
`parsePropsPattern`, which now shares it. `findStatementEnd` got the
trailing `,`/`=` ASI check via `lastMeaningfulBefore`.

Two old pins updated on purpose: the accented-identifier test (it used to
register the garbage stem `caf`; now it registers nothing) and the
destructuring/multi-declarator pins, which now assert the support. Tests
cover renames, nesting, defaults, rest, mixed declarator lists, multi-line
lists across trailing commas, glued `const{a}`, imports inside
initializers, and the end-to-end reactive render (`Component79.test.ts`).
`docs/setup-scripts.md` replaced its "not supported" paragraph with the
supported forms.
